## PROPOSAL (Step 4 — one enhancement)
- **Type:** process/tool — a new **defense/scoring instrument** (the programmatic CLD scorer). Additive,
  reversible, non-structural: it *implements* the already-governed CLD structural-grading method (Structure
  §3.1 line 94 / §4.6 line 156) — it authors no new rule, weight, or invariant.
- **Target dimension:** A (system representation — feedback loops, polarity, delays), the construct CLD tests;
  it does not alter the A–E composite weights.
- **Diff summary:** two staged artifacts, moved live at APPLY:
  - `items/cld_oracle.json` — machine-readable encoding of the 5 SenseRun-#7 reference solutions (canonical
    variables · required signed edges · loop inventory as ordered cycles · dominant loop · key delays · the
    named trap). The validation set.
  - `engine/cld-score.py` — the scorer. `score <oracle> <ITEM_ID> <resp.json>` parses a model's structured
    CLD output `{variables, edges(signed), loops(R/B + node cycle), dominant_loop, names_dominant_shift,
    delays}`, **recomputes every declared loop's polarity as the product of its signed edges** and checks it
    against both the model's own R/B label and the oracle, then scores: `base = 0.50·edges + 0.30·loops +
    0.20·variables`; `final = base` if the dynamic insight (dominant-loop shift / driving delay) is named,
    `min(base, 0.5)` if the structure is right but the insight is missing (the item-file partial-credit rule),
    `min(base, 0.25)` if the dominant loop is R↔B mislabelled (the named trap → low score). `calibrate
    <oracle>` is a fail-closed self-test (mirrors `jury-stats.py`).
- **Expected effect:**
  - *Gradeability (the chosen lever):* CLD's deterministic structural sub-score becomes **machine-executable**
    — Design-Review category 3 moves from "specified but manual" toward "deterministic oracle."
  - *Scoring redundancy:* CLD becomes a true **second *executable* judge-independent path** end-to-end
    (serves ANTIPATTERNS #2 "single path to a number").
  - *Cost:* removes the manual/subjectivity bottleneck in CLD structural grading (Meadows #6 information-flow).
  - *Validation:* the 5 reference solutions become a running test set; the #7 hand-verification (loop sign =
    product of edges, 11/11) is now reproduced **by code** on every `calibrate`.
  - *Discrimination (designed-in, not measured):* the trap cap (≤0.25 for a dominant-loop R↔B mislabel) and the
    insight gate (0.5 without the dominant-shift/delay) are what separate genuine structural insight from
    loop-naming — empirical IRT `a` still needs a future live-model run.
- **Honest non-certification:** the CLD **completeness/quality** (jury) sub-score is **unchanged** — still
  `UNCALIBRATED — not scored` (no CLD gold; fail-closed §5.1). The scorer emits **no jury number**. The
  component weights (0.50/0.30/0.20) and the trap cap (0.25) are **tool-internal policy implementing the
  item-file prose, not governance** — tunable, and flagged in RECURSE for later calibration against real
  responses. No IRT/discrimination is measured (no live spend).
- **Rollback plan:** the APPLY is one git commit via `engine/apply-commit.sh 8`; `engine/rollback-run.sh 8`
  reverts exactly it (removes the scorer + oracle, restores INDEX/version/CHANGELOG/Research-ledger).
  Reversible by construction.
- **Pre-REVIEW validation (staged):** `python3 engine/cld-score.py.staging calibrate

## THOUGHT PROCESS (the narrative — why this move, what we expect, what we're watching)
The benchmark's binding *additive* constraint had moved exactly where SenseRun #7 predicted it would. #7
seeded the CLD format and its structural-oracle specs but stopped short of building the grader — that would
have been a second lever, and untested without items. So #7 left the dominant additive idea (BACKLOG #2, the
programmatic CLD scorer) *feasible for the first time* but unbuilt, with the next-run seed leaning explicitly
toward building it. Re-sensing this run confirmed the gap was real and unmoved: the §3.1 CLD structural method
("loop sign = product of signed edges") existed only as **prose** in the item file; grading a CLD answer meant
a human applying that spec by hand. The §8.1 self-audit put Gradeability at 3 for exactly this reason — half
the deterministic surface (the newest format) was manual. Discrimination (2) is lower but unreachable in a
SenseRun (it needs live-model response data, which the rail forbids), so the dominant *feasible* lever was
Gradeability, and the dominant feasible move was the scorer.

The scorer is a Meadows-#6 (information-flow) intervention: it doesn't change what CLD measures or how the
construct is weighted; it removes the manual/subjectivity bottleneck that stood between a governed scoring
method and a running number. I was careful that this is *implementation*, not rule-making — Structure §3.1
and §4.6 already mandate that CLD's structural part be "scored programmatically … loop-polarity check,
reference-label match." Building the tool authors no new rule, touches no A–E weight, and leaves the parked
§3.1(b) accept-rule hardening (the genuine Meadows #5) firmly with August + Ember. The one place I introduced
numbers the prose didn't fix — the component weights (0.50/0.30/0.20) and the trap cap (0.25) — I treated as
tool-internal policy, documented as tunable and flagged for calibration against real responses, not as
governance. The 0.5 partial-credit cap is taken verbatim from the item file.

I built the scorer in the established house idiom: subcommands plus a fail-closed `calibrate` self-test that
must reproduce known-true numbers before any output is trusted — exactly how `jury-stats.py` self-calibrates
against the recorded gold arithmetic. That self-test is also the no-regress evidence: it reproduces #7's
11/11 loop-sign verification *by code* and round-trips every reference solution to 1.0. A grader that can't
re-derive its own validation set is not trustworthy; this one does, 31/31.

What we expect: CLD's deterministic structural sub-score is now machine-executable end-to-end — a true second
judge-independent path, the way SF is, directly answering the standing "single path to a number" anti-pattern.
What we are watching: (1) whether real model outputs parse cleanly into the structured schema or whether a
node-alias layer is needed (the scorer normalizes case/space/punctuation and tolerates reverse-direction edge
lookups and full-word "Reinforcing"/"Balancing", but free-form variable names from a live model may still
miss the canonical set — flagged to backlog); (2) whether the trap cap (≤0.25 for a dominant-loop R↔B
mislabel) and the insight gate (0.5 without the dominant-shift/delay) actually separate genuine structural
insight from loop-naming once a live run produces a score distribution; (3) whether the component weights need
recalibration against that distribution. What we did NOT certify, stated plainly: no CLD score is
jury-validated (completeness stays `UNCALIBRATED — not scored`); no IRT discrimination is measured (no live
spend); the scorer's policy constants are reasonable defaults, not empirically tuned. And the parked §3.1(b)
hardening (Meadows #5) and Ember's pending #5 countersign were not touched — both remain the council's call.
